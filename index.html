03

Situación Problemática

Problem Statement

Context / Contexto

Context

Santa Cruz Pharmacy manages $12,000 worth of inventory annually. Its manual record-keeping system has caused severe operational and financial problems including product losses and inefficient processes.

The pharmacy operates in a competitive market and needs to modernize its operations to remain viable and efficient.

Contexto

La Farmacia Santa Cruz gestiona anualmente $12,000 en inventario. Su sistema manual de registro ha causado severos conflictos operativos y financieros, incluyendo pérdidas de productos y procesos ineficientes.

La farmacia opera en un mercado competitivo y necesita modernizar sus operaciones para permanecer viable y eficiente.

Identified Problems

Financial Problems

  • $400 annual losses from expired products
  • Inability to track cost price changes
  • Delayed pricing adjustments
  • No inventory valuation reports

Problemas Financieros

  • $400 pérdidas anuales por productos vencidos
  • Incapacidad de rastrear cambios en costos
  • Ajustes de precios tardíos
  • Sin reportes de valoración de inventario
Identified Problems II

Operational Problems

  • Difficulty tracking stock levels
  • Slow product location (manual search)
  • No expiration date alerts
  • Decreased customer satisfaction

Problemas Operativos

  • Dificultad para rastrear niveles de stock
  • Búsqueda lenta de productos (manual)
  • Sin alertas de fechas de vencimiento
  • Disminución de satisfacción de clientes
Causes and Effects / Causas y Efectos

Root Causes

  • Manual inventory system
  • No digital records
  • Lack of automation tools
  • No data analysis capability

Causas Raíz

  • Sistema manual de inventario
  • Sin registros digitales
  • Falta de herramientas de automatización
  • Sin capacidad de análisis de datos

Solution Overview

The Solution

A digital inventory management system that automates processes, tracks products, generates alerts, and provides real-time reports for decision-making.

El sistema está construido usando tecnologías web modernas con una interfaz fácil de usar que requiere mínimo entrenamiento.

Requisitos del Sistema

System Requirements

Requisitos Funcionales

¿Qué son los Requisitos Funcionales?

Los requisitos funcionales describen las funciones y características que el sistema debe realizar. Definen qué debe hacer el software desde la perspectiva del usuario.

Para el sistema de inventario de Farmacia Santa Cruz, estos requisitos definen las operaciones que el sistema realizará: autenticación, gestión de productos, ventas, compras, alertas y reportes.

Functional Requirements (1-4) / Requisitos Funcionales

RF-001: Inicio de Sesión

El sistema debe permitir el acceso mediante usuario y contraseña.

Prioridad: ALTA | Actores: Administrador, Vendedor

RF-002: Registrar Productos

Permitir el registro de nuevos productos con nombre, precio, stock, laboratorio, fecha de vencimiento y categoría.

Prioridad: ALTA | Actor: Administrador

RF-003: Registrar Compras

Registrar la compra de productos a proveedores, actualizando el stock y el precio de costo.

Prioridad: ALTA | Actor: Administrador

RF-004: Registrar Ventas

Registrar ventas, calcular total, generar factura y descontar stock automáticamente.

Prioridad: ALTA | Actor: Vendedor

Functional Requirements (5-7) / Requisitos Funcionales

RF-005: Monitoreo de Costos

Detectar cambios en costo de compra y sugerir actualización de precio de venta.

Prioridad: MEDIA | Actor: Administrador

RF-006: Alerta de Caducidad

Generar alertas para medicamentos próximos a vencer.

Prioridad: ALTA | Actor: Administrador

RF-007: Generar Reportes

Generar reportes de inventario, ventas y vencimientos.

Prioridad: MEDIA | Actor: Administrador

Requisitos No Funcionales

¿Qué son los Requisitos No Funcionales?

Los requisitos no funcionales definen las cualidades y restricciones del sistema. Describen cómo debe funcionar el sistema, no qué debe hacer.

Incluyen aspectos como rendimiento, seguridad, usabilidad, disponibilidad, mantenibilidad y compatibilidad.

Non-Functional Requirements (1-4) / Requisitos No Funcionales

RNF-001: Rendimiento

  • Tiempo de respuesta < 2 segundos
  • Soporte para 10 usuarios simultáneos
  • Capacidad de 5,000+ productos

RNF-002: Seguridad

  • Contraseñas encriptadas (SHA-256)
  • Control de acceso por roles
  • Sesiones con timeout de 30 min

RNF-003: Usabilidad

  • Interfaz intuitiva y fácil de usar
  • Mínimo entrenamiento requerido
  • Diseño responsivo (móvil y PC)

RNF-004: Disponibilidad

  • Disponibilidad 99% del tiempo
  • Recuperación ante errores
  • Respaldo de datos diario
Non-Functional Requirements (5-6) / Requisitos No Funcionales

RNF-005: Mantenibilidad

  • Código modular y documentado
  • Fácil actualización de precios
  • Extensible para futuras funciones

RNF-006: Compatibilidad

  • Compatible con Chrome y Edge
  • Funciona sin conexión a internet
  • Base de datos SQLite/MySQL

Technology Stack

Technologies to Use

Flutter (Frontend)

Google's framework for cross-platform development. Creates apps for Android, iOS, and Web from a single codebase. Ideal for native performance and large community.

• Language: Dart
• UI: Material Design
• State: Provider/Riverpod

PHP (Backend)

Server-side scripting language widely used for web development. Ideal for connecting to SQL databases and creating dynamic applications.

• Language: PHP
• Frameworks: Native
• Server: Apache

MySQL (Database)

Relational database management system that uses SQL language to organize information in tables. Stores data in structured tables with relationships between them.

• Tables & Relations
• SQL Queries
• Data Integrity

Why This Choice?

Reliable, well-documented technology, local data storage without internet dependency, and proven for small to medium applications.

• Local storage
• No internet needed
• Easy maintenance

MySQL (Base de Datos)

Sistema de gestión de bases de datos relacionales que utiliza el lenguaje SQL para organizar información en tablas. Almacena datos en tablas estructuradas con relaciones entre ellas.

• Tablas y Relaciones
• Consultas SQL

¿Por qué esta elección?

Tecnología confiable y bien documentada, almacenamiento local de datos sin dependencia de internet, y probada para aplicaciones pequeñas y medianas.

• Almacenamiento local
• No requiere internet
• Fácil mantenimiento

Arquitectura de la Solución

Arquitectura de la Solución

Modelo Local: La aplicación se ejecuta en un servidor local, almacenando los datos en una base de datos SQL en el mismo dispositivo o en red local.

Flujo de datos: Flutter → PHP API → MySQL/SQLite. Las consultas se realizan a través de endpoints PHP que interactúan con la base de datos.

Seguridad: Control de acceso por roles, validación de datos en servidor, y conexión segura a la base de datos.

Conclusion / Conclusión

Conclusion

This module taught us to develop a system's structure without programming, focusing on planning and organization. We worked on use case diagrams, sequence diagrams, class diagrams, ER diagrams, interfaces, and data dictionaries.

This project strengthened teamwork and addresses key problems: inventory losses, inefficient searches, lack of reports, and stock management.

Conclusión

Este módulo nos enseñó a desarrollar la estructura de un sistema sin programarlo, centrándonos en su planificación y organización. Trabajamos en diagramas de casos de uso, secuencias, clases, ER, interfaces y diccionarios de datos.

Este proyecto fortaleció el trabajo en equipo y aborda problemas clave: pérdidas de inventario, búsquedas ineficientes, falta de reportes y dificultades en la gestión de stock.
